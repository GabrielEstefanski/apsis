# v0.1.0-alpha.1 — first pre-release checkpoint

**Date:** 2026-05-02
**Status:** Pre-release (alpha). Public API stabilised but not tagged final; pre-release scope remains open.
**Tag:** `v0.1.0-alpha.1`
**Branch:** merged into `master` via PR #25 from `develop`.
**License:** Apache License 2.0.

This is the first version snapshot of `apsis`, marking the point at which the federation thesis is implemented end-to-end with a complete reference downstream consumer (`apsis-1pn` / `apsis-1pn-py`) and the cross-implementation parity portfolio against REBOUND IAS15 covers four canonical regimes.

> **Post-release note (2026-05-15).** The Python distribution has since been consolidated: `apsis-1pn-py` is folded into the `apsis.gr` submodule of the unified `apsis` distribution, and `apsis-py` becomes `apsis-python` (PyO3 cdylib backing the same `import apsis` surface). Install instructions and import statements below describe the v0.1.0-alpha.1 tagged state; current usage is `pip install apsis` + `from apsis.gr import PostNewtonian1PN`.

The release closes no scope. The `alpha` marker is deliberate: WH refactor, octree, optimisations, material-system review, and `apsis-app` plugin features are all candidates for inclusion before `v0.1.0` final, and the documentation does not predict which of those will land where.

---

## Highlights

- **Federation thesis end-to-end.** A force perturbation is published as an independent Cargo crate, declares its physical preconditions on the gravitational kernel as type-level `KernelRequirements`, and is composed into a simulation through the library's public extension API. `apsis-1pn` is the reference downstream consumer.
- **3D-aware physics core.** `Vec3` value type, per-body `z` / `vz`, 3D-aware direct N-body kernel, IAS15 dense interpolation in 3D, orbital elements with inclination $i$ and node $\Omega$, 3D observables (energy, angular momentum, centre-of-mass).
- **`apsis::contract` formalised as executable specification.** Twelve CI tests covering kernel invariants, composition rules, and the failure model, co-located with the prose statement of each guarantee.
- **Cross-implementation parity portfolio against REBOUND IAS15.** Four canonical scenarios spanning periodic 2-body, periodic 3-body, chaotic 3-body, and sign-flipped 2-body regimes (including a $10^4$-orbit long-horizon retrograde Kepler). All gated invariant metrics agree at 1 ULP of f64 machine epsilon in regime.
- **Mercury perihelion precession reproduced to ~1 ppm of GR.** Over 500 orbits under IAS15 + 1PN with the default exact `NewtonKernel` (ε = 0). Asserted in CI at a 100-ppm threshold that absorbs cross-platform LLVM / libm variance.
- **Python binding with researcher-first API.** `apsis-py` exposes `Body`, `IntegratorKind`, `System`, `Trajectory`, and adaptive-controller diagnostics through kwargs — designed for scientists, not for thin C-API exposure.

## What this release contains

### Core library — `apsis`

The library implements direct $O(N^2)$ Newtonian gravitation in 3D Cartesian coordinates, with optional Plummer softening, `UnitSystem` as the locked physics-units contract (G derived from $G_\text{SI} \cdot M \cdot T^2 / L^3$), and a structured-diagnostic event bus surfacing kernel-precondition mismatches at perturbation registration. Templates ship for canonical scenarios (Solar System, TRAPPIST-1, figure-eight, Pythagorean, Pluto–Charon, hot Jupiter, Sun–Earth–Moon, equal-mass binary, and others).

The `apsis::contract` module formalises three guarantee classes — kernel invariants, composition rules, and the failure model — as types and CI-asserted tests. Reading the module top-to-bottom reads the contract; running `cargo test -p apsis --lib contract` verifies it.

### Integrators

| Integrator | Order | Status |
| --- | --- | --- |
| Velocity Verlet | 2nd-order symplectic | Production |
| Yoshida 4 | 4th-order symplectic composition | Production; default for interactive playback (see [ADR-003](../adr/003-integrator-execution-profile.md)) |
| IAS15 | 15th-order adaptive Gauss–Radau | Production; spec-conformant against Rein & Spiegel (2015) §2–3, used for all validation runs |
| Wisdom–Holman | mixed-variable | Carries four documented algorithmic defects (TD-008); informational only, not treated as a quality signal in validation runs |

The IAS15 implementation has undergone an architecture audit against the algorithmic specification, documented in [`docs/experiments/2026-04-26-ias15-warmstart-bug.md`](../experiments/2026-04-26-ias15-warmstart-bug.md). A subsequent velocity-prediction bug discovery and fix (R&S 2015 eq. 11) is documented in [`docs/experiments/2026-04-28-ias15-velocity-prediction-bug.md`](../experiments/2026-04-28-ias15-velocity-prediction-bug.md); that fix moved Mercury's residual error from a 4.4 ppm systematic bias to ~1 ppm stochastic round-off.

### Federation: plugin protocol and reference implementation

| Crate | Role |
| --- | --- |
| [`apsis-py-core`](../../crates/apsis-py-core/) | Cross-extension transport (rlib): `Box<dyn PerturbationForce>` ↔ `PyCapsule`. Allows out-of-tree force crates to expose Python bindings without re-implementing physics. |
| [`apsis-1pn`](../../crates/apsis-1pn/) | First downstream force crate: Einstein–Infeld–Hoffmann (Schwarzschild limit) 1PN correction. Reference implementation of the federation contract. Depends only on `apsis` through the public extension API. |
| [`apsis-1pn-py`](../../crates/apsis-1pn-py/) | Python binding for `apsis-1pn`. Reference implementation at the Rust ↔ Python boundary. Includes an inclined Mercury smoke test ($i = 7°$) exercising the 3D path. |

### Python binding — `apsis-py`

PyO3-based, abi3-py39. Researcher-first kwargs API exposing:

- `Body`, `IntegratorKind`, `System` factories with explicit kwargs (no positional pitfalls);
- `Trajectory` with `System.sample(times=...)` for NumPy-friendly recording (positions, velocities, energy as 2-D arrays);
- `Stats` and `AdaptiveStats` for adaptive-controller diagnostics;
- `position` and `velocity` accept either 2-tuples (implicit $z = 0$) or 3-tuples; planar code keeps working unchanged.

### Validation portfolio

**Cross-implementation parity vs REBOUND IAS15** (four scenarios, all in [`validation/rebound-parity/`](../../validation/rebound-parity/)):

| Scenario | Regime | Horizon | Worst observed | Notebook |
| --- | --- | --- | --- | --- |
| Kepler-prograde ($e = 0.5$) | periodic 2-body, $L_z > 0$ | 100 orbits | 1–3 ULP on 7 gated metrics | [2026-04-25](../experiments/2026-04-25-rebound-parity-kepler.md) |
| Figure-8 (Chenciner–Montgomery) | periodic 3-body, $L_z = 0$ | 10 + 50 (informational) periods | 1 ULP on 12 gated metrics | [2026-04-26](../experiments/2026-04-26-rebound-parity-figure8.md) |
| Pythagorean (Burrau 1913) | chaotic 3-body | 70 canonical t.u. | structural invariants ($\mathbf{L}, \mathbf{P}, \mathbf{r}_\text{COM}$) at f64 floor on both sides; energy bound exceeded symmetrically (regime mismatch with smooth-flow bound, not a parity defect); 98 % event alignment | [2026-04-30](../experiments/2026-04-30-rebound-parity-pythagorean.md) |
| Kepler-retrograde ($L_z < 0$) | sign-flipped 2-body | $10^4$ orbits + 100-orbit checkpoint | 1–10 ULP on 10 metrics × 2 horizons | [2026-05-01](../experiments/2026-05-01-rebound-parity-retrograde.md) |

The retrograde long-horizon scenario closes the long-horizon stability gate identified during the GR-readiness review as a precondition for 1PN-class extensions.

**Heuristic validation** ([`paper/notebooks/2026-05-01-recommended-dt-validation.md`](../../paper/notebooks/2026-05-01-recommended-dt-validation.md)):

`recommended_dt` (the heuristic that `System` returns when asked "what timestep should I use?") was validated across 13 scenarios × 3 fixed-step integrators. 26/26 gated cells pass under the standard isclose-style two-sided Tier 2 bound. A per-cell utilisation metric ($u = \text{peak}/\text{bound}$) is emitted as a regression canary; the only "tight" cell across the grid is `hd_80606_b_system` VV at $u = 0.668$, which is mechanism-explained in §Interpretation as the high-eccentricity close-encounter regime stretching VV's smooth-flow assumption.

**Mercury perihelion precession** ([`crates/apsis-1pn/tests/mercury_precession_gate.rs`](../../crates/apsis-1pn/tests/mercury_precession_gate.rs)):

The library's headline analytic-physics validation. Sun + Mercury under IAS15 + 1PN with the default exact `NewtonKernel` (ε = 0), integrated for 500 orbital periods. Measured perihelion drift: $+42.983$ arcsec/century vs the closed-form GR prediction $+43$ arcsec/century, a relative error of $\sim 1$ ppm on developer hardware (at the f64 noise floor of the test-particle 1PN approximation). CI gate at 100 ppm absorbs cross-platform variance.

## How to install / try it

The full Quickstart lives in the [README §Quickstart](../../README.md#quickstart). In brief:

```bash
git clone https://github.com/GabrielEstefanski/apsis && cd apsis
# Python binding (after the first PyPI release this becomes pip install apsis apsis-1pn)
pip install maturin
maturin develop --release --manifest-path crates/apsis-py/Cargo.toml
maturin develop --release --manifest-path crates/apsis-1pn-py/Cargo.toml
```

Then in Python:

```python
import apsis
import apsis_1pn

sun = apsis.Body.star(mass=1.0)
mercury = apsis.Body.rocky(
    mass=1.66e-7, position=(0.387, 0.0), velocity=(0.0, 1.61),
)

sys = apsis.System(
    bodies=[sun, mercury], units=apsis.units.SOLAR,
    integrator="ias15", dt=1e-3,
)
sys.add_perturbation(apsis_1pn.PostNewtonian1PN.solar_units())
sys.integrate_for(100.0)
```

For a runnable Rust example with the GR comparison printed inline:

```bash
cargo run --release --example mercury_perihelion -p apsis-1pn
```

## Currently evolving

The `alpha` marker reflects that several subsystems are candidates for change before `v0.1.0` final. Areas under active consideration include the Wisdom–Holman integrator refactor (TD-008), Barnes–Hut octree implementation, performance optimisations, the material-system review, and `apsis-app` plugin-feature integration. None of these is committed to a specific subsequent version; this release does not predict the boundary between subsequent alpha checkpoints and the `v0.1.0` final tag.

The federation thesis itself, the parity portfolio, the `apsis::contract` surface, and the Mercury 1PN result are stable and will not regress.

## Known limitations

- **Wisdom–Holman carries four documented algorithmic defects (TD-008)** and is not treated as a quality signal in validation runs.
- **Validated regime is currently $N \le 10^3$.** Larger-$N$ behaviour is not covered by the parity portfolio or the unit-test surface.
- **Single-precision interactive trails.** The interactive-shell trail buffer stores positions as `f32` for GPU efficiency. This affects only the visual history; integrator state is `f64` throughout.
- **IAS15 per-step cost is unbounded in stiff regimes** (intrinsic to adaptive high-order integrators). Driving IAS15 from a render loop at high $N$ is not recommended; see [ADR-003](../adr/003-integrator-execution-profile.md).
- **Out of current scope:** symplectic compositions beyond Yoshida-4, MERCURIUS-style close-encounter switching, stellar evolution, hydrodynamics, collisionless large-N.

## Reproducibility

The Mercury 1PN gate reproduces on a clean checkout with a single command after installing a Rust 1.85+ toolchain:

```bash
cargo test --release -p apsis-1pn --tests -- --ignored
```

Each parity scenario in the portfolio reproduces via its self-contained Python harness (REBOUND 4.6.0 required):

```bash
python validation/rebound-parity/kepler/run.py
python validation/rebound-parity/figure8/run.py
python validation/rebound-parity/pythagorean/run.py
python validation/rebound-parity/retrograde/run.py
```

The `recommended_dt` validation reproduces via two pure-Rust commands:

```bash
cargo run --release --example recommended_dt_validation -p apsis
cargo run --release --example recommended_dt_compare -p apsis
```

Continuous-integration configuration: `cargo fmt --all`, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo test --workspace`, plus a core-isolation gate (`apsis` resolves no UI dependency) and the release-mode Mercury gate.
